01 / Brief
TeamClaw Memory Direction
For developers

TeamClaw 的长期记忆系统

目标不是“做一个万能大记忆库”，而是做一个 workspace-scoped、file-first、支持 personal + team 的 local agent memory system。

Workspace First 所有记忆暂时只存在于当前工作目录内，不跨 workspace。
Prompt Discipline 默认只注入少量高价值记忆，其余通过检索召回。
Team Sync 共享记忆通过 teamclaw-team/ 目录走 P2P 或 OSS。
Why now: make the agent improve with repeated use

02 / Problem
Why memory now

我们缺的不是 RAG，缺的是统一的 memory model

Today

已有能力

  • workspace RAG 与 hybrid search
  • team sync: `teamclaw-team/` + P2P / OSS
  • skills、session history、knowledge/memory 入口
Gap

缺失的统一规则

  • 哪些信息该沉淀为 personal memory
  • 哪些信息该进入 team memory
  • 哪些记忆默认注入，哪些只做检索
  • 新旧结论冲突时如何保留历史
Main pain: durable knowledge currently has no home
Key priorities: 历史决策 / 调试经验 / 项目约定
03 / Benchmark
Hermes vs OpenClaw vs TeamClaw

我们最像什么

Hermes Agent

分层很强

  • `MEMORY.md` / `USER.md` 是高价值长期记忆
  • skills 是程序性记忆
  • sessions 用 DB / JSONL，不和长期记忆混在一起
OpenClaw

file-first 很强

  • Markdown 是 source of truth
  • 长期记忆与 daily memory 分层
  • hybrid search + recency / decay 很适合本地 agent
TeamClaw Target

OpenClaw 骨架 + Hermes 分层

  • workspace-first、file-first
  • raw episodes / personal / team / derived index 四层
  • 额外支持 `teamclaw-team/` 共享 scope
Conclusion: not graph-first, not DB-first
Direction: inspectable files + derived retrieval
04 / Model
Layered memory model

四层模型

Raw Episodes

聊天、tool 输出、报错、测试结果、关键代码变更。是来源材料，不默认注入。

Personal Memory

私有且 durable 的 workspace 记忆，允许 agent 主动写入。

Team Memory

共享给当前 workspace 团队，存放在 `teamclaw-team/knowledge/memory/`。

Derived Index

向量、BM25、rerank、prompt cache。它们是派生物，不是 source of truth。

Most important boundary: source files vs derived index
This keeps rebuilds, sync, and debugging simple
05 / Scope
What gets remembered

优先沉淀的三类记忆

Kind 01

历史决策

为什么采用某方案、为什么放弃某方案。要保留历史，支持 `active / superseded`。

Kind 02

调试经验

定位过程、根因、修复手法。大多走检索，不适合大量常驻 prompt。

Kind 03

项目约定

命名、测试、目录、流程约束。最适合进入 always-on memory。

默认 scope

先写入 personal；只有明显对多人有价值时才写入 team。

来源必须足够丰富

不只看聊天，还要看代码变更、tool 结果、错误、测试输出。

Memory kinds align with the product’s real value
Not all session data deserves promotion
06 / Prompt
How memory enters the prompt

两段式 prompt 注入

系统提示词中的稳定块

只放少量高价值、当前有效的记忆。建议 `5-12` 条，控制在 `300-800 tokens`。

每轮动态替换的记忆块

根据当前任务检索相关 memory。适合调试经验、旧决策细节、历史上下文。

Raw Episodes 不常驻

原始 episode 只作为检索源或 consolidation 原料，不直接塞进系统 prompt。

[Workspace Memory - Always On] Team conventions: - All new backend endpoints require tests. - P2P membership treats manifest as authoritative. Active decisions: - Shared assets live in teamclaw-team/. Personal: - Prefer concise developer-facing summaries. [Relevant Memory Retrieved For This Task] - [team][debugging][active] Watcher re-entry caused duplicate indexing; debounce fixes it. - [team][decision][superseded] Older reconnect logic trusted local state more than manifest.
Prompt discipline matters more than memory volume
Convention: always / Decision: mixed / Debugging: retrieve
07 / Write Rules
How proactive writes stay sane

允许主动写，但必须带护栏

Write when
  • 明确说“记住这个”
  • 一个方案已经被采纳或废弃
  • 一个 bug 根因已验证
  • 重复工作形成了稳定约定
Do not write when
  • 只是猜测，尚未验证
  • 只是临时状态或一次性上下文
  • 明显是个人偏好，却被误判成 team rule
  • 高时效信息很快就会过期

Every memory has provenance

记录 session、file、tool、error、test 等来源。

Every memory has status

`active / superseded / stale`，避免静默覆盖历史。

Every team write is reversible

冲突时保留旧记录，当前结论成为新版本。

Auto-write, not auto-chaos
Main safeguard: provenance + status + supersedes
08 / Layout
Where the files live

建议的目录布局

<workspace>/ ├── .teamclaw/ │ └── memory/ │ ├── episodes/ │ ├── personal/ │ │ ├── conventions/ │ │ ├── decisions/ │ │ └── debugging/ │ └── index/ └── teamclaw-team/ └── knowledge/ └── memory/ ├── conventions/ ├── decisions/ └── debugging/

Source of truth

可读可改的 memory files。适合 sync、review、diff、回滚。

Derived only

index 可以删掉重建，不应成为唯一真相源。

Simple file layout beats opaque state
This also matches the current TeamClaw architecture better
09 / Decision
What v1 should be

v1 应该像什么，不应该像什么

Should be
  • workspace-scoped
  • file-first source of truth
  • small always-on memory + strong retrieval
  • personal/team scope split
  • history-preserving conflict model
Should not be
  • a giant prompt blob
  • a graph-first system in v1
  • a DB-only black box
  • manual-only memory curation
  • cross-workspace memory too early

一句话结论

OpenClaw 风格的 file-first memory 作为骨架，借鉴 Hermes 的分层思路，再补上 TeamClaw 的 teamclaw-team 共享 scope。

Recommended next step: schema + retrieval contract
Enough direction for implementation discussion
